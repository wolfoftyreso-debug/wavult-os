What it is

Compliance that actually works.

ISO certification is won with a consultant, a binder of procedures, and weeks of prep before the external audit. Then everyone goes back to working the way they always have. The binder collects dust. The next audit cycle starts from scratch.

Process is different because it's not a separate compliance layer. It's connected to Execution, DMS, and every other module. When a task is overdue, Process knows. When a vehicle handover deviates from the checklist, Process knows. When a document version is superseded, old versions are locked automatically. The compliance record builds itself in real time — not the week before the audit.

The result: external auditors have never seen documentation this complete. Internal teams don't experience compliance as overhead — because it happens as a side effect of doing the work.

Core features

What Process does.

⚠️

Non-Conformances with State Machine

Every non-conformance moves through a strict lifecycle. Severity is assigned at creation (Minor, Major, Critical) and determines escalation rules. An NC cannot skip states — you cannot close what hasn't been analyzed and actioned.

Each state requires specific fields to advance: ANALYZING requires a root cause entry; ACTION requires at least one corrective action with an owner and deadline; CLOSED requires a verification sign-off. Critical NCs require manager approval to close.

Technical

NC states are enforced server-side via a transition guard system. Each transition validates required fields and permission levels. Transitions emit nc.state_changed webhook events. State history is immutable — every transition is an append-only log entry with timestamp and actor ID.

📋

ISO Standards Compliance Tracking

pixdrift ships with requirement maps for ISO 9001, 14001, and 45001. Each clause maps to specific controls — document types, process steps, and evidence requirements. As you work in the system, requirements are marked as evidenced automatically.

Before an audit, the compliance dashboard shows: which clauses are fully evidenced, which need attention, and what evidence exists for each requirement. An export generates a structured evidence package formatted for external auditors.

Technical

ISO requirement sets are versioned data models (iso_standard_version entities). Each requirement has a control_type (document, process, record) and links to evidence_items generated by platform activity. Compliance score = evidenced requirements / total requirements × 100.

📄

Document Control with Version Management

Documents in Process have a formal lifecycle: Draft → Review → Approved → Obsolete. Only the current approved version is accessible by default. Older versions are archived, not deleted — they're accessible to auditors but not surfaced to users.

Document access is controlled by role. Approval workflows are configurable: some documents need one approver, others need a chain. Changes require a change reason. Expiry dates trigger review reminders automatically.

Technical

Document versions use semantic versioning (major.minor). Major versions require approval workflow; minor versions require the document owner's acknowledgment. Files are stored in immutable object storage with content-addressed URIs. Metadata is signed on approval using server-side HMAC-SHA256.

Technical

Risk thresholds are defined as JSON policy rules attached to the organization's Process configuration. Escalation actions are configurable per threshold: notify, create_task, create_nc, or webhook. Aligns with ISO 31000 risk management principles.

🔒

Immutable Audit Trail

Every action across the entire platform is recorded in the audit trail: who did what, when, and what changed. Records are append-only — no updates, no deletes. Content hashes detect any tampering attempt.

The audit trail is exportable as a signed PDF or structured JSON for external auditors. ISO external auditors can be granted read-only access directly — no need to prepare documentation packages manually.

Technical

Audit entries are written to an append-only PostgreSQL partition with SECURITY DEFINER functions preventing direct row modification. Each entry includes a SHA-256 hash of its content plus the previous entry's hash, forming a chain. Monthly chain integrity checks run automatically and alert on any break.

🔍

Inspection Protocols for Automotive & Industry

Pre-built inspection protocol templates for automotive (pre-delivery inspection, annual service, vehicle handover) and industrial environments. Custom protocols are built with the form builder — checkbox lists, measurement inputs, photo capture, and digital signature.

Completed protocols are linked to the relevant entity (vehicle, work order, deal) and stored in the immutable record. Failed inspection items automatically generate NCs with the protocol as source evidence.

Technical

Protocols are defined as JSON schema templates with field types: checkbox, measurement, photo, signature, text. Submitted responses are validated against the schema and stored as immutable protocol_response entities. Photo evidence is stored as content-addressed files with SHA-256 integrity hashes.

Module integrations

How Process connects to everything else.

Compliance is most powerful when it's automatic — when work itself generates the compliance record, rather than a separate task.

Task Overdue → NC Created An overdue task in Execution opens an NC in Process with severity based on task priority. NC links back to the task as source.
Inspection Fail → NC + DMS Note A failed item in a vehicle inspection protocol creates an NC and adds a note to the vehicle's DMS record simultaneously.
Document Approved → ISO Evidence Each document approval automatically creates an evidence entry linked to the relevant ISO requirement clauses.
NC Closed → Capability Feedback When an NC root cause links to a skill gap, Capability module receives a development recommendation for the involved team member.
Risk Escalated → Execution Task Critical risks above threshold automatically create a task in Execution assigned to the risk owner, with deadline and NC if unaddressed.
API Reference

Build on Process.

The Process API is fully RESTful. All write operations are atomic and emit webhook events. The audit trail is readable via API but never writable.

Endpoint Method Description
/v1/ncs POST Create non-conformance. Required: title, severity, source_entity_type, source_entity_id.
/v1/ncs/{id}/transition POST Transition NC state. Body: { "to": "ANALYZING", "root_cause": "..." }. Returns 422 if required fields missing.
/v1/ncs GET List NCs. Filter by state, severity, source_entity_id, assignee_id. Paginated.
/v1/documents POST Create document draft. Upload file via multipart. Returns document with version 0.1 and status DRAFT.
/v1/documents/{id}/approve POST Approve document. Advances to APPROVED if approval workflow satisfied. Emits document.approved event.
/v1/audit-trail GET Read audit trail. Filter by entity_type, entity_id, actor_id, date range. Read-only. Cannot be written via API.
/v1/risks POST Create risk. Required: title, probability (1–4), impact (1–4), owner_id.
