What it is

Operations management without the overhead.

Execution is where your business happens. Sales reps track deals. Operations managers assign tasks. Leadership reviews KPIs. And every decision — from pricing exceptions to client escalations — gets logged with a timestamp and a name.

Most companies cobble this together: a CRM here, a task tool there, a shared Notion doc for decisions, a Slack thread for customer conversations. Execution replaces all four — and adds something none of them have: a live connection to what's actually happening in your business.

When a task goes overdue, it's not just a notification. It creates a non-conformance in the Process module. When a deal closes, Finance already knows — an invoice drafts automatically. That's not an integration. That's a single system that understands context.

Core features

What Execution does.

📊

Deal Pipeline with Probability Tracking

Deals move through configurable stages from first contact to closed. Each stage carries a win probability that rolls up to a team forecast. Automated follow-up reminders fire when deals go quiet — configurable per stage, per rep, or per deal value.

Every interaction is logged chronologically: calls, emails, meetings, and notes. Filtering by deal source, deal size, or assigned rep takes one click. Pipeline views are configurable per role — a rep sees their own pipeline; a manager sees the full team.

Technical

Deals are modeled as a directed acyclic graph (DAG) of stages. Stage transitions are validated server-side and emit webhook events. Win probabilities are user-defined per stage and aggregate to a weighted pipeline value. All transitions are immutable audit log entries.

✅

Task Management with Hard Enforcement

Tasks in Execution are not suggestions. Each task can have completion conditions — a form that must be filled, a document that must be uploaded, an approval that must be received. Until those conditions are met, the task simply cannot be marked complete. No workarounds.

Tasks are assigned to individuals or teams, with deadlines, priority levels, and linked entities (deals, contacts, work orders). Overdue tasks escalate automatically based on configurable rules.

Technical

Task completion uses a state machine: PENDING → IN_PROGRESS → BLOCKED → REVIEW → DONE. Transitions are guarded by completion conditions evaluated server-side. An overdue task (past deadline with status ≠ DONE) automatically emits a task.overdue event that can trigger NC creation in Process module.

📝

Decision Log with Audit Trail

Every decision that matters gets logged: who made it, when, what the options were, and what was chosen. Decisions are linked to the entities they affect — a deal, a process, a team member, a client. Six months later, when someone asks "why did we do this?", you have the answer.

The log is immutable. Entries cannot be edited or deleted — only supplemented with follow-up entries. This creates a legally defensible record of business decisions.

Technical

Decision entries are append-only records stored with SHA-256 content hashes for tamper detection. Each entry carries: author_id, timestamp (UTC, microsecond precision), entity_type, entity_id, decision_body, and an optional rationale field. Full-text indexed via Elasticsearch.

📈

Team KPI Dashboards per Role

A sales rep's dashboard looks different from a CFO's. Execution ships with role-specific dashboard templates that surface the metrics that matter for each role — and lets you customize from there.

Dashboards pull live data from across modules: deal velocity from Execution, budget variance from Finance, open NCs from Process, skill gaps from Capability. One screen, real context.

Technical

Dashboard widgets are reactive components subscribing to a server-sent event (SSE) stream. Data is aggregated via materialized views refreshed on write, not on query — keeping dashboard loads under 100ms regardless of data volume. Custom metrics can be defined via a JSON DSL.

🗓️

Meeting Management with Protocols

Meetings in Execution have a before, during, and after. Before: agenda is set. During: notes are taken against agenda points. After: action items are automatically converted to tracked tasks with assignees and deadlines. Meeting protocols are stored, linked to the relevant deal or project, and searchable.

Technical

Action items extracted from meeting notes use a structured markup format (@assign[user], @due[date]) that the parser converts to Task entities. Meetings link to calendar events via iCal/Google Calendar webhook integration.

💬

Channels — Context-Linked Communication

Every deal, project, and task has its own communication channel. Conversations are linked to the work — not buried in a general Slack channel or lost in email threads. When a new team member joins a deal, they get the full context instantly.

Channels support file attachments, @mentions, and links to other entities. Notifications are configurable per user and per channel type. No context switching to a separate tool.

Technical

Channels use a fan-out pub/sub architecture. Messages are persisted to PostgreSQL and delivered via WebSocket. Each channel has a context_entity field linking it to a Deal, Task, Process, or any other entity. Read receipts are tracked per user per message.

Module integrations

How Execution connects to everything else.

This is what separates pixdrift from a collection of tools. When something happens in Execution, the right thing happens everywhere else — automatically, without configuration for common scenarios.

Deal Won → Finance · Invoice Invoice draft created with deal value, customer data, and line items pre-filled. Accountant reviews and sends.
Task Overdue → Process · NC A non-conformance is opened in Process with severity level based on task priority and overdue duration.
Deal Note: Skill Gap → Capability · Development When a deal note tags a competence need, Capability module receives a development suggestion for the assigned rep.
Payout Approved → Banking · Transaction Approved payout triggers a payment initiation via PSD2 Open Banking connection. Reconciliation is automatic.
Meeting Action Item → Execution · Task Action items from meeting protocols become tracked tasks with owners and deadlines — inside the same module.
API Reference

Build on Execution.

The Execution API follows REST conventions. All endpoints require Bearer token authentication. Responses are JSON. Webhook events deliver state changes in real time.

Endpoint Method Description
/v1/deals GET List deals with filter, sort, and pagination. Supports stage, assignee_id, probability_min filters.
/v1/deals POST Create deal. Required: title, contact_id, stage_id. Returns deal with generated ID and audit log entry.
/v1/deals/{id}/stage PATCH Advance deal stage. Validates allowed transitions. Emits deal.stage_changed webhook on success.
/v1/tasks GET List tasks. Supports assignee_id, status, due_before, entity_type filters.
/v1/tasks/{id}/complete POST Attempt task completion. Returns 409 if completion conditions not met, with list of unmet conditions.
/v1/decisions POST Log decision. Required: body, entity_type, entity_id. Entry is immutable once created.
/v1/decisions GET Search decision log. Full-text search via q param. Filterable by entity_id, author_id, date range.
