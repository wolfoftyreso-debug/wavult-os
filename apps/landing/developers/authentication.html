Authentication

pixdrift uses Bearer token authentication. Every API request must include an Authorization header.

Quick start: Create an API key in your pixdrift app under Admin → Settings → API Keys, then include it as Authorization: Bearer pd_live_<key>.

API Keys

API keys are the recommended authentication method for server-to-server integrations. They are long-lived, scoped, and can be revoked individually.

Key format

pd_live_a1b2c3d4e5f6... (64 hex chars after prefix)

Creating a key

Via the API (requires existing auth):

curl -X POST https://api.bc.pixdrift.com/api/admin/api-keys \
  -H "Authorization: Bearer YOUR_JWT" \
  -H "Content-Type: application/json" \
  -d '{"name": "My Integration", "scopes": ["read", "write"]}'

# Response:
{
  "id": "uuid",
  "name": "My Integration",
  "key": "pd_live_abc123...",  ← Store this — shown only once!
  "scopes": ["read", "write"],
  "created_at": "2026-03-21T10:00:00Z"
}
Security: The raw key is shown only once at creation. Store it in a secret manager (AWS Secrets Manager, Vault, etc.). The server only stores a SHA-256 hash.

Using a key

curl https://api.bc.pixdrift.com/api/contacts \
  -H "Authorization: Bearer pd_live_abc123..."

Listing & revoking keys

# List keys (hashes never returned, only metadata)
curl https://api.bc.pixdrift.com/api/admin/api-keys \
  -H "Authorization: Bearer YOUR_TOKEN"

# Revoke a key
curl -X DELETE https://api.bc.pixdrift.com/api/admin/api-keys/KEY_ID \
  -H "Authorization: Bearer YOUR_TOKEN"

JWT Tokens (Supabase)

For user-facing apps, use Supabase JWT tokens obtained from the client-side auth flow. These are short-lived (1 hour) and tied to a specific user context.

// JavaScript — Supabase client
import { createClient } from '@supabase/supabase-js'
const supabase = createClient(SUPABASE_URL, SUPABASE_ANON_KEY)

const { data: { session } } = await supabase.auth.getSession()
const token = session.access_token  // Use as Bearer token

Scopes

API keys can be restricted to specific scopes:

ScopeDescription
readGET requests on all resources
writePOST, PATCH, PUT on resources
deleteDELETE requests
adminAdmin operations (user mgmt, API keys)
reportsAccess to all reports and exports
enterpriseElevated rate limits (1000 req/15min)

Rate Limits

Rate limits apply per API key in a sliding 15-minute window.

PlanLimit
Standard100 requests / 15 min
Growth300 requests / 15 min
Enterprise (scope)1,000 requests / 15 min

Rate limit headers on every response:

X-RateLimit-Limit: 100
X-RateLimit-Remaining: 87
X-RateLimit-Reset: 1711015200  # Unix timestamp when window resets

When exceeded, you receive 429 Too Many Requests. Retry after the X-RateLimit-Reset timestamp.

Webhook Signature Verification

Webhook payloads are signed with HMAC-SHA256. Always verify the signature before processing.

// Node.js
const crypto = require('crypto')

function verifyWebhook(payload, signature, secret) {
  const expected = crypto
    .createHmac('sha256', secret)
    .update(payload)
    .digest('hex')
  return crypto.timingSafeEqual(
    Buffer.from(expected),
    Buffer.from(signature)
  )
}

See Webhooks documentation for full details.

CORS

CORS is configured to allow all origins by default (*). For production, set CORS_ORIGIN environment variable to your domain. Browser requests to the API will work without a proxy.

Access-Control-Allow-Origin: https://app.yourcompany.com
Access-Control-Allow-Methods: GET, POST, PUT, PATCH, DELETE, OPTIONS
Access-Control-Allow-Headers: Authorization, Content-Type
